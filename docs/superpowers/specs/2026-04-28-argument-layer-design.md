# Argument Layer for Per-Section Drafting (Phase 1)

**Date:** 2026-04-28
**Status:** Spec — pending implementation plan
**Owner:** zak@sunstone.ie

## Problem

The report-builder pipeline drafts each section as prose given a `load_bearing_finding` (a single-sentence goal), a research bundle, and a facts catalog. The agent decides on its own how to structure the argument that establishes the load-bearing finding. Quality control is downstream and lexical: DeSlop catches AI-tells, design-deslop catches headlines that drift toward AI-flavor. Neither catches the deeper failure mode of AI-written research: **prose that reads beautifully but does not actually establish what the load-bearing finding claims it does.**

Symptoms include sentences with quantitative content unsupported by any fact, sub-arguments that contradict each other across paragraphs, methodological caveats that get glossed over or omitted, and a sense at the end of a section that the prose has *gestured at* the finding rather than *established* it.

The fix is to make the argument structure of each section a first-class artifact — generated, persisted, edited, and consumed by the drafter — rather than something the drafter improvises in prose.

## Goal

Insert a new pipeline phase between research and drafting that produces an **argument graph** for the section: a small tree of claims, with the section's conclusion at the root, supporting sub-claims as internal nodes, and grounding facts at the leaves. The graph is generated by an AI agent grounded in the just-completed research bundle, persisted onto the section, optionally edited by the user, and consumed by the drafter as its primary structural input.

The drafter's job changes from "write 800 words about the load-bearing finding" to "render this argument as prose."

## Non-goals (Phase 1)

- **Kickoff intent capture.** The kickoff agent is not modified to ask the user about expected sub-claims. The argument graph is generated cold from research output. (Phase 2 concern.)
- **Auto-validation of prose against the graph.** No new pipeline phase reads the final prose and flags sentences that don't map to a claim. The argument graph is a drafting input only, not a quality gate. (Phase 1.5 concern, to be designed once we see real prose coming out of the new flow.)
- **Cross-section argument validation.** No `report check` command that walks inter-section dependencies between graphs. (Phase 3 concern.)
- **Report-level argument graph.** Phase 1 is per-section only. The relationship between sections' argument graphs is not modeled.
- **Hard approval gate.** No new `SectionStatus` values, no `report accept-argument` command. The graph is reviewed via file inspection and edited via direct YAML editing.

## Design

### Pipeline placement

Per-section lifecycle today (post body-acceptance design stages omitted for clarity):

```
PENDING → DRAFTING → AWAITING_ACCEPT → ACCEPTED
[run_pipeline: research → drafting → synthesis → deslop → AWAITING_ACCEPT]
```

Per-section lifecycle with the argument layer (Phase 1):

```
PENDING → DRAFTING → AWAITING_ACCEPT → ACCEPTED

[run_pipeline:
   research
     ↓
   propose-argument                                           ← NEW phase
     ↓
   (soft pause: pipeline returns; section.argument persisted)
     ↓
   (user reviews `report show <id> --argument`,
    optionally edits sections/<id>.yaml directly,
    runs `report next <id>` to resume)
     ↓
   drafting (consumes argument graph)                         ← MODIFIED
     ↓
   synthesis → deslop → AWAITING_ACCEPT
]
```

### Soft gate semantics

`run_pipeline` becomes resumable: on every invocation, it inspects section state and advances from the next unblocked phase. `section.brief` is set unconditionally at the top of every invocation (existing behavior, idempotent), so `brief` is not a useful branching signal. The branching uses `research_notes`, `argument`, and `drafts` instead.

```
on entry:
  always: set section.brief from outline_entry, save section.

  if section.research_notes is None:
      run research; persist research_notes (and any new facts/charts).
      attempt propose-argument; persist argument on success.
      return  (pause for user review of argument graph)

  elif section.argument is None:
      # Research ran on a prior invocation but propose-argument failed.
      # Don't re-run research — retry just the argument step.
      attempt propose-argument; persist argument on success.
      return  (pause for user review)

  elif section.drafts == []:
      # Argument is set, drafts are empty — resume from drafting.
      skip research and propose-argument.
      run drafting (with argument graph injected) → synthesis → deslop → AWAITING_ACCEPT.

  else:
      # Drafts already exist — this is a re-run after prose rejection or
      # an `report argue` regeneration that overwrote the prior argument.
      run drafting (existing rerun path: rejection_history feeds the prompt) →
        synthesis → deslop → AWAITING_ACCEPT.
```

Re-invoking `report next <id>` always advances to the next unblocked phase. The user's only action between research and drafting is reviewing/editing `section.argument` in the YAML; if they regenerate via `report argue <id>`, drafts (if any) stay in place but will be overwritten on the next drafting run.

No new `SectionStatus` values are introduced. Section status remains `DRAFTING` while paused after argument is proposed; the section yaml carrying `argument` populated and `drafts` empty is the implicit pause-state marker.

### Schema additions

In `report_builder/schemas.py`:

```python
class Claim(_Base):
    """A single node in a section's argument graph.

    Leaf claims have empty depends_on and reference grounding facts via
    supports_via. Internal claims have non-empty depends_on; they may also
    reference facts directly. Caveats are leaf claims that the prose must
    acknowledge as constraints rather than establish as findings.
    """
    id: str                                       # "c1", "c2", ...
    text: str
    depends_on: list[str] = Field(default_factory=list)
    supports_via: list[str] = Field(default_factory=list)   # fact ids
    is_caveat: bool = False


class ArgumentGraph(_Base):
    """The argument structure for a single section.

    conclusion_id points at the claim that is THE conclusion — the realized
    version of the section's load_bearing_finding. The drafter writes prose
    that establishes the conclusion through the supporting claim structure.
    """
    conclusion_id: str
    claims: list[Claim]
```

`Section` gains two optional fields:

```python
class Section(_Base):
    # ... existing fields preserved ...
    research_notes: Optional[str] = None        # persisted bundle.notes for resume
    argument: Optional[ArgumentGraph] = None    # the proposed/edited argument graph
```

Both default to `None` so existing section yaml files load without migration. `_Base.extra="ignore"` continues to handle forward-compat.

### Validation invariants

The following invariants on `ArgumentGraph` are checked at load time (warnings) and after `propose_argument` produces a graph (hard fail with retry):

- `argument.conclusion_id` must match some `claim.id` in `argument.claims`.
- Every id in `claim.depends_on` must reference another `claim.id` in the same graph.
- The graph (claims as nodes, `depends_on` as edges) must be acyclic.
- `claim.supports_via` ids should exist in `facts.yaml`. Missing facts are surfaced as warnings on load (facts can be renamed/removed independently between sessions); on initial generation they are a hard validation failure that triggers a retry.

A graph that fails validation after the retry budget is rejected and the user is asked to re-run `report argue <id>`.

### New components

**`report_builder/argument.py`** — single-purpose module mirroring `design_pack.py` in shape.

Public API:

```python
@dataclass
class ProposeArgumentResult:
    graph: Optional[ArgumentGraph]
    error: Optional[str] = None


def propose_argument(
    project: ReportProject,
    section: Section,
    research_notes: str,
    facts_mgr: FactsManager,
    *,
    agent_call: Optional[Callable[..., dict]] = None,
    max_validation_retries: int = 2,
) -> ProposeArgumentResult:
    ...
```

Implementation:
- Reads system prompt from `report_builder/prompts/argument_system.md`.
- Builds user prompt with section brief, load_bearing_finding, research notes, facts catalog, voice constraints.
- Calls Anthropic (or stub for tests).
- Parses JSON output, validates against schema and the invariants above, retries with corrective hint on validation failure (mirrors `design_pack`'s pull-quote retry loop).
- Returns `ProposeArgumentResult` with the parsed graph or an error string.

**`report_builder/prompts/argument_system.md`** — system prompt with:
- The role: produce an argument graph that establishes the section's load-bearing finding.
- Hard rules: every claim must be either grounded in a fact (via `supports_via`) or build on other claims (via `depends_on`); the conclusion claim's text should be a clear realization of the load-bearing finding (it may refine wording but must not change scope); methodological caveats are flagged with `is_caveat: true`.
- Output JSON schema matching the `ArgumentGraph` model.

### Modifications to existing files

**`report_builder/pipeline.py`** — `run_pipeline` updated with the resume logic above. New phase: between research (phase 1) and drafting (phase 2), invoke `propose_argument` if `section.argument is None`, persist result, return early with informational log.

**`report_builder/drafters.py`** — `_build_user_prompt` adds one block between facts and stylesheet:

```
# Argument structure (render this as prose; do not introduce claims outside it)

<rendered graph from render_argument_graph(section)>
```

Plus one sentence appended to the prompt's instructions: *"The argument structure above is your structural backbone. Render it as prose. Do not introduce quantitative or definitive claims that aren't in the argument graph."*

**`report_builder/memory.py`** — new helper `render_argument_graph(section: Section) -> str`. Produces a tree-view text rendering of the graph: conclusion at top; supporting claims in dependency order with their grounding facts inline; caveats grouped at the bottom.

**`report_builder/cli.py`** — two surface changes:
- `report show <section-id>` accepts a new `--argument` flag that pretty-prints the graph (tree view, same renderer used in the drafter prompt).
- New command `report argue <section-id>` — explicit regenerate. Calls `propose_argument` again, overwrites `section.argument`. Errors with a helpful message if `section.research_notes is None` (research hasn't run).

**`report_builder/schemas.py`** — additions described above.

### Drafter behavior change

The drafter receives the argument graph as a structural directive in its user prompt. The expected behavioral change:

- Drafts follow the conclusion → sub-claim → fact backbone explicitly, in dependency order.
- Methodological caveats are acknowledged as constraints, not buried.
- Quantitative claims appear in prose only when grounded by a `supports_via` fact reference.
- Drafts do not introduce arguments outside the graph (e.g., editorial asides about the broader implications) unless they're added to the graph first.

The existing rejection-feedback flow (`section.rejection_history`) continues to operate on the prose level. Argument-level rejection is not formalized in Phase 1; if the user dislikes the argument structure, they edit the YAML or run `report argue <id>` to regenerate.

### Edge cases

**Empty research bundle.** If research returns no facts and no notes, `propose_argument` errors out with: *"insufficient research output to propose an argument; run research again or pre-populate facts.yaml."* Section stays at status DRAFTING with `brief` set, `research_notes` empty, `argument` None.

**User edits create an invalid graph.** On the next `report next <id>`, the load-time validation surfaces warnings via `report show` but does not block drafting. The drafter consumes whatever the YAML says. (Rationale: the user knows their own intent better than schema validation can; warning rather than blocking respects that. Hard validation only runs at AI-generation time, not at load time.)

**User regenerates argument after drafting has produced drafts.** `report argue <id>` is allowed in this state; it overwrites `section.argument` but does NOT clear `section.drafts`. Re-running `report next <id>` afterwards will see drafts non-empty and re-run drafting (consistent with the existing rerun path), now consuming the new argument graph. This is the correct behavior: if you regenerated the argument, you almost certainly want to re-draft.

**`research_notes` desync after manual edits to facts.yaml.** Persisted research notes might reference facts that were later renamed or removed by hand. On resume, the drafter consumes the stored notes verbatim — it does not re-validate them against the current facts catalog. If a fact rename is severe enough to invalidate the research narrative, the user runs `report next <id>` from a section reset to PENDING (the existing recovery path).

### Testing

Unit tests:
- `Claim` and `ArgumentGraph` round-trip serialization (mirrors existing `test_schemas_design.py`).
- `argument.propose_argument` with stub agent: produces a valid graph; validation catches missing conclusion_id; validation catches dangling depends_on references; validation catches cycles; retry loop re-prompts on validation failure with corrective hint; gives up after max retries.
- `render_argument_graph` produces stable tree-view output for a fixed graph.

Integration test:
- Run a fixture project through the full new flow with stubbed agents: research → propose-argument → soft-pause → resume → drafting → synthesis → deslop. Assert: `section.argument` populated after first invocation; `section.drafts` empty after first invocation; `section.drafts` populated after second invocation; argument graph rendered into the drafter's prompt (verifiable via inspecting a prompt snapshot or instrumenting the drafter in test mode).

## Open questions

None at spec time. Implementation plan will resolve remaining detail (specific retry budget, exact tree-view rendering format, JSON schema for the agent's output).

## Follow-ups (out of scope)

- Phase 1.5: prose-vs-graph auto-validation (LLM-based or regex-based "every quantitative sentence must reference a fact" check, surfaced as warnings before AWAITING_ACCEPT).
- Phase 2: kickoff intent capture and proposal-against-intent diff (`origin: confirmed_from_intent | refined_from_intent | emerged_from_research` per claim).
- Phase 3: cross-section argument validation (`report check` walks dependencies between sections' graphs and surfaces missing supports).
- Argument-graph-aware rejection: `report reject --claim c2 --reason "..."` that targets a specific claim rather than the prose as a whole.
- Multi-renderer output: produce executive memo, board deck, technical appendix from the same argument graph + facts (decoupling analysis from prose rendering).
